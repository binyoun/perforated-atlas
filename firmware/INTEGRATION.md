# Perforated Atlas — Version 2 Firmware Integration Contract

Target: ESP32 driving a stepper motor (paper strip transport), a WS2812 LED
(or strip), and a speaker/solenoid comb. This document is the complete
contract between the web engine and the physical machine. If something is
ambiguous here, ask before improvising — the schema is locked.

**Privacy invariant: the original visitor text never exists anywhere in the
system. You will only ever receive the pattern below.**

---

## 1. Strip JSON schema

Every strip — whether fetched from the archive or pre-generated — is one
JSON object:

```jsonc
{
  "id": "a3f1c9e2-...",         // UUID v4, string
  "created_at": "2026-07-05T02:14:00.000Z", // ISO8601
  "locale_hint": "vi",           // "vi" | "ko" | "en" | "unknown"
  "source_length": 42,           // char count of the ORIGINAL text (count only)
  "city": "Hồ Chí Minh",         // may be ""
  "country": "Vietnam",          // may be ""
  "word_count": 7,
  "notes": [ /* NoteEvent[], see below */ ],
  "palette": ["#e8b25f", "#5f8fe8"], // two dominant hex colors
  "strip_length_mm": 380         // physical length: round(totalDuration * 36)
}
```

Each element of `notes` is a `NoteEvent`:

```jsonc
{
  "t": 1.25,        // onset, seconds from strip start
  "pitch": "C5",    // one of the 30 Sankyo pitches, or null for a rest
  "dur": 0.25,      // duration in seconds
  "vel": 0.8,       // velocity 0–1 (0 for rests)
  "hue": 42         // HSL hue 0–360, or null for rests
}
```

Guarantees you can rely on:

- `notes` is sorted by `t`, and events are contiguous: `notes[i+1].t == notes[i].t + notes[i].dur`.
- Total duration in seconds = sum of all `dur` = `strip_length_mm / 36`.
- Rests always have `pitch: null`, `vel: 0`, `hue: null`. Skip them for
  sound/LED; they still consume transport time (the paper keeps moving).
- `pitch` is always one of the 30 strings in the table in section 3.
- The physical scroll speed is **36 mm/s** (locked).

## 2. Fetching strips from Supabase (plain REST, no SDK)

The archive is a single Postgres table `strips` behind Supabase's REST API
(PostgREST). Read access uses the **anon key** — no auth flow needed.

```
GET {SUPABASE_URL}/rest/v1/strips?select=*&order=created_at.desc&limit=24
Headers:
  apikey: {SUPABASE_ANON_KEY}
  Authorization: Bearer {SUPABASE_ANON_KEY}
```

Fetch one strip by id:

```
GET {SUPABASE_URL}/rest/v1/strips?id=eq.{uuid}&select=*
```

Both return a JSON array of StripJSON rows (the second returns an array of
one). Example with curl:

```bash
curl -s "$SUPABASE_URL/rest/v1/strips?select=*&order=created_at.desc&limit=1" \
  -H "apikey: $SUPABASE_ANON_KEY" \
  -H "Authorization: Bearer $SUPABASE_ANON_KEY"
```

On ESP32 (Arduino/ESP-IDF) this is a single HTTPS GET with two headers.
Budget for response size: a 42-char memory is ~45 note events ≈ 4–5 KB of
JSON. Parse with ArduinoJson (`DynamicJsonDocument` ≥ 16 KB to be safe) or
stream-parse the `notes` array.

Offline fallback: strips can be pre-generated by running the pure engine
(`src/engine/translate.ts`) in Node 18+ and flashing the JSON to SPIFFS/SD.
The engine is DOM-free and dependency-free by design.

## 3. Pitch → MIDI note number (complete lookup table)

Sankyo 30-note layout, canonical, index 0 = lowest. `Db` = C#, `Eb` = D#,
`Gb` = F#, `Ab` = G#, `Bb` = A#.

| 0  | C4  | 60 | 261.63 | | 15 | Eb5 | 75 | 622.25 |

| 5  | F4  | 65 | 349.23 | | 20 | Ab5 | 80 | 830.61 |

Note the top octave is diatonic-ish (C6 D6 E6 F6 G6 A6) — MIDI numbers jump
by 2. As C code:

```c
// Indexed by SANKYO_30 order (0 = C4 ... 29 = A6)
static const uint8_t SANKYO_MIDI[30] = {
  60,61,62,63,64,65,66,67,68,69,70,71,      // C4..B4
  72,73,74,75,76,77,78,79,80,81,82,83,      // C5..B5
  84,86,88,89,91,93                          // C6 D6 E6 F6 G6 A6
};
```

## 4. Time → stepper motor steps

The transport advances the strip at a constant physical speed of 36 mm/s.
Define one hardware constant, calibrated on the assembled machine:

```c
// Steps the motor must make for the strip to travel for exactly 1 second
// (i.e. 36 mm of paper). Depends on motor steps/rev, microstepping, and
// roller diameter. CALIBRATE ON HARDWARE.
#define STEPS_PER_SECOND  /* e.g. 720 */
```

Then for every note event:

```c
steps_offset   = note.t   * STEPS_PER_SECOND;  // absolute position of onset
steps_duration = note.dur * STEPS_PER_SECOND;  // length of the hole/rest
total_steps    = total_duration_s * STEPS_PER_SECOND;
```

Derivation of the calibration constant, if you want a starting value:
`STEPS_PER_SECOND = (motor_steps_per_rev × microstep) × 36 / (π × roller_diameter_mm)`.
Run the motor at a constant step rate; never modulate speed by content —
tempo is carried entirely by the hole spacing.

## 5. Hue → WS2812 RGB

Each sounding note carries `hue` (0–360). The installation LED color is
`hsl(hue, 60%, 50%)`. Convert HSL→RGB on-device:

```c
// h: 0-360, returns 8-bit RGB for hsl(h, 60%, 50%)
void hue_to_rgb(float h, uint8_t *r, uint8_t *g, uint8_t *b) {
  const float s = 0.60f, l = 0.50f;
  float c = (1.0f - fabsf(2.0f * l - 1.0f)) * s;   // = 0.6
  float hp = fmodf(h, 360.0f) / 60.0f;
  float x = c * (1.0f - fabsf(fmodf(hp, 2.0f) - 1.0f));
  float rf = 0, gf = 0, bf = 0;
  if      (hp < 1) { rf = c; gf = x; }
  else if (hp < 2) { rf = x; gf = c; }
  else if (hp < 3) { gf = c; bf = x; }
  else if (hp < 4) { gf = x; bf = c; }
  else if (hp < 5) { rf = x; bf = c; }
  else             { rf = c; bf = x; }
  float m = l - c / 2.0f;                           // = 0.2
  *r = (uint8_t)((rf + m) * 255.0f);
  *g = (uint8_t)((gf + m) * 255.0f);
  *b = (uint8_t)((bf + m) * 255.0f);
}
```

Rests (`hue == null`): LED off, or hold the previous color decaying — the
web version decays each glow over ~1.1 s (ease-out, exponent 1.6); match
that feel if you implement afterglow.

`palette` (two hex colors per strip) is available for ambient/idle lighting
between notes or between strips.

## 6. Velocity → LED brightness

```c
uint8_t brightness = (uint8_t)(note.vel * 255.0f);  // vel is 0–1
```

Currently the engine emits `vel: 0.8` for every sounding note and `0` for
rests, so treat velocity as forward-compatible headroom — scale, don't
hardcode 204.

## 7. Suggested playback loop (reference)

1. Fetch or load StripJSON; parse `notes`.
2. Precompute `steps_offset` per note (section 4).
3. Start stepper at constant rate. Track absolute step count.
4. When the step count crosses a note's `steps_offset` and `pitch != null`:
   - trigger the comb/speaker at `SANKYO_MIDI[pitch_index]`
   - set the LED to `hue_to_rgb(note.hue)` at `vel * 255`
5. When the strip ends (`total_steps`), stop, idle-breathe the LED
   (the web version pulses at a 3 s period, 0.8–1.0 scale), await the next
   visitor or strip.
